## 2. 対象環境（確定事項）
### 2.1 マシン構成
- Mac: **Apple Silicon** / **持ち運び可能**
- Linux: **Ubuntu** / **NVIDIA GPUあり** / **持ち運び不可** / **常時稼働可能**

### 2.2 ネットワーク条件
- Linux は **自宅LAN**
- Mac と Linux は **同じネットワークに繋がらない**（外出先などからアクセスしたい想定）

### 2.3 利用目的の比率
- **開発支援**と**文書作成**は **半々**

## 3. mubo の概要（コンセプト）
- `mubo` は **One-shot local LLM bootstrapper**（ローカルLLM実行環境を `setup.sh` 一発で自動構築するブートストラッパ）。
- 主なフロー（README/specの要旨）:
  1. 環境検出（OS/CPU/RAM/GPU等）
  2. Ollama 導入（未導入なら）+ 疎通確認
  3. ベースモデル `gpt-oss:20b` を pull
  4. 環境メモリに合わせた **長コンテキスト派生モデル**を作成
  5. extras（vLLM/MLX/uv/Dockerなど）を条件が合えば導入（失敗してもコアは壊さない）
  6. Web UI の自己改変エージェントをデプロイ（ブラウザから操作）

---

## 4. Ollama 導入済みでも mubo を使う「価値」
### 4.1 価値の中心
- **Ollamaの再インストール**が目的ではなく、次を自動で揃えるのが価値：
  - Ollama サーバ起動/疎通確認（`localhost:11434` 前提）
  - `gpt-oss:20b` の pull（未取得なら）
  - 派生モデル（デフォルト名: `gpt-oss:20b-long`）を `ollama create` で作る
  - エージェント Web UI（デフォルト `http://localhost:8392`）

### 4.2 派生モデル（長コンテキスト）のポイント
- `setup.sh` はメモリ状況により `num_ctx` を自動決定（16K/32K/64K/128K）。
- `Modelfile` を生成し、`ollama create` で派生モデルを作る。
- これにより「自分の環境で現実的に動くコンテキスト長」を毎回悩まずに済む。

---

## 5. mubo エージェント（Web UI）の要点（運用で触る設定）
### 5.1 エージェントの接続先 / モデル
- 環境変数で切替可能（ローカル運用なら基本デフォルトでOK）:
  - `OLLAMA_BASE_URL`（デフォルト: `http://localhost:11434`）
  - `MUBO_MODEL`（デフォルト: `gpt-oss:20b-long`）

### 5.2 ポート
- `MUBO_PORT`（デフォルト: `8392`）
- 複数起動したい場合はポートをずらせる

---

## 6. 2台運用（“2人のエージェント”）の基本方針
### 6.1 共通の土台
- **両マシンで `setup.sh` を1回通す**ことで、以下の共通土台を作る：
  - Ollama 疎通OK
  - `gpt-oss:20b` + `gpt-oss:20b-long`（環境最適 `num_ctx`）を整備
  - Web UI（8392）でエージェント利用可能

### 6.2 役割分担（提案）
#### Mac（Apple Silicon / 持ち運び）
- “フロント担当（フィールドエージェント）”
- 即応・対話・要約・設計・タスク分解・指示書作成（Linuxへ投げる実行パケット生成）

#### Linux（Ubuntu / NVIDIA / 常時稼働）
- “バックヤード担当（重作業ワーカー）”
- 長時間生成、重い推論、反復検証、将来的な高スループット構成（extrasのvLLMなど）を試す母艦

---

## 7. Linux（NVIDIA）側で重要な観点：Docker + GPU 到達性チェック
- `setup.sh` は Docker の導入・デーモン起動確認を行い、
  - Linux + NVIDIA の場合に **nvidia-docker の機能テスト**を実施
  - `docker run --gpus all ... nvidia-smi` が通るかを確認
  - GPU がコンテナから見えるなら `NVIDIA_DOCKER_OK=true` のように状態を保持
- これが通る環境は、将来の拡張（コンテナ前提の推論基盤）にも載せやすい。

---

## 8. “Mac→Linuxへ渡す” ための接続課題と解決方針（SSH）
### 8.1 現状の課題
- Linux は自宅LAN、Macは同一ネットワークにいないため **通常のLAN内SSHでは届かない**。

### 8.2 取れる選択肢（整理）
1. **Tailscale（推奨）**
   - ポート開放不要、NAT越えを吸収しやすい
   - 外出先Macから自宅Ubuntuへ “仮想LAN” として接続できる
2. ルータの **ポート開放**
   - SSHをインターネットに晒すためリスク高（非推奨寄り）
   - 実施するなら鍵ログイン必須等の堅牢化が必要
3. 踏み台（VPS等）/ リバースSSH
   - 構成としては強いが手順がやや複雑

### 8.3 推奨方針（決定済みではなく提案）
- 条件から **Tailscale が最も適合**（自宅LAN・同一NW不可・常時稼働可）。

---

## 9. 2人エージェント運用のワークフロー案（提案）
### 案A：司令（Mac）→ 実行（Linux）→ 統合（Mac）
1. Mac エージェントで目的/制約/入力を整理
2. Mac エージェントが「Linuxに貼る実行パケット」を作成
   - 目的 / 期待アウトプット形式 / 制約 / 反復方針（例：3案→自己評価→改善）
3. Linux エージェントで重い生成・反復を実行
4. 結果をMacへ戻し、最終編集・意思決定

### 案B：並列→統合
- Mac：設計・評価軸作り・要約・編集
- Linux：各章ドラフト生成・大量比較・長時間処理
- Mac：統合して成果物化
