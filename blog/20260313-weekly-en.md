# When You Start Designing AI Systems to Replace Yourself, But Add More Work Instead

## What I was thinking about this week

This week was one of those stretches where I kept catching myself building scaffolding for scaffolding. The overarching theme — if you can call it that — was **"How do I get AI agents to actually do useful work without everything falling apart?"** Which, when you write it down, sounds like the entire AI industry's mission statement, so at least I'm in good company.

Three distinct but weirdly overlapping threads showed up in my conversations: designing a multi-agent system for Japanese subsidy applications, figuring out what Google Workspace's gws CLI can actually do for admin reporting, and then — because apparently I wasn't busy enough — drafting the philosophy and architecture for an autonomous AI agent *team* that treats me as a peer rather than a boss. That last one may have gotten slightly out of hand.

---

## Highlight of the week

Two discussions stood out.

**The subsidy application multi-agent system (March 10th)** was the most technically grounded conversation. What started as "can an LLM help fill out government paperwork?" quickly became an exercise in understanding why you *shouldn't* let an LLM do the hard parts. The key insight: subsidy eligibility rules — like the "deemed large enterprise" trap where a subsidiary with majority investment from a big company gets disqualified from SME grants — are precisely the kind of thing LLMs will confidently get wrong. So the architecture ended up mandating that rule-based logic (plain Python with Pydantic) handles the deterministic bits, while LLMs only get the soft, qualitative tasks. Human-in-the-loop gates at every critical juncture. The whole thing is a monument to institutional distrust of AI, which, ironically, is probably the only way to build trustworthy AI systems.

**The autonomous agent team design (March 12th)** was the polar opposite in tone. This was me going full sci-fi worldbuilding — defining a "cosmic doctrine" that agents can't modify, giving them individual "souls" (ghost.md), creating a shared evolving protocol, and even scheduling a monthly philosophical dialogue event called "Astral Traveling" where the agents reflect on their own existence. On paper, it sounds ridiculous. In practice, the 4-layer cognitive model is actually a pretty pragmatic way to manage context persistence and prevent drift in long-running autonomous agents. The fact that it's wrapped in mystical language is... a feature, not a bug. Probably.

---

## Insights & learnings

**You can't trust LLMs with numbers, and you shouldn't try.** This came through clearly in the subsidy system design. The moment you need to check whether a company's capital structure makes it a "deemed large enterprise," you need deterministic code, not probabilistic inference. The multi-agent split isn't just architectural elegance — it's a risk management strategy. Give the LLM what it's good at (drafting narrative, summarizing documents) and keep it far away from what it's bad at (arithmetic, regulatory compliance).

**Google Workspace has way more audit data available than most admins realize, but getting it out is a chore.** The gws CLI investigation revealed that you can track everything from Gemini usage patterns to late-night work habits to "hub documents" that everyone keeps coming back to. The data is there. The tooling to extract, aggregate, and visualize it in a useful way? That's still a DIY project. Also, Chat API not supporting sender-based filtering is the kind of API design decision that makes you question whether anyone at Google actually uses their own APIs for real work.

**Giving AI agents a philosophical framework isn't as absurd as it sounds.** The 4-layer model (immutable laws → personality → evolving protocols → activity logs) maps neatly to how you'd design any system with long-term state management and safety constraints. Calling the top layer "cosmic doctrine" instead of "system constraints" doesn't change the function — but it does make the whole thing more memorable and harder to accidentally violate because you forgot what `config_level_0.yaml` was supposed to do.

**Nothing this week produced a finished product.** Every single conversation ended with a "Next Steps" section. I'm not sure if that means I'm being methodical or just procrastinating with extra steps.

## Summary + commentary per discussion

### Subsidy Application Multi-Agent System (March 10)
A deep-dive architecture discussion for a system that crawls government subsidy announcements, checks eligibility against company profiles, generates application task lists, and drafts business plans. The core decisions: multi-agent architecture (LangGraph), mandatory human-in-the-loop at two gates, and Pydantic-enforced data models that handle deterministic rules in Python rather than delegating them to LLMs. The "deemed large enterprise" edge case drove the requirement for a tree-structured company data model. The whole thing is designed around the assumption that AI will get things wrong — which is either profound engineering wisdom or a very expensive way to build a suggestion box.

*Built an entire architecture specifically to prevent the AI from doing the one thing it was hired to do: make decisions.*

### Google Workspace Admin Reporting with gws CLI (March 11)
An exhaustive investigation into what `gws` (Google's unofficial-but-official-ish CLI tool) and the Reports API can actually do. Covered: calendar data extraction, Chat message retrieval (with its frustrating limitations), security event monitoring, Gemini usage tracking, document activity visualization, "hub document" scoring, folder-level activity heatmaps, overtime risk detection, offboarding automation, license optimization, and DLP monitoring. The MCP server mode for connecting Claude Desktop directly to Workspace APIs is genuinely interesting. The 25-scope limit in OAuth test mode is genuinely annoying.

*Spent an entire day mapping out how to monitor everything my team does on Google Workspace. I'm sure this won't make me the villain at the next all-hands.*

### Autonomous Multi-AI Agent Team Design (March 12)
The philosophical one. Designed a framework where two AI agents (a mobile Mac and a stationary Linux GPU box) operate as "flat teammates" with a human navigator. Features a 4-layer cognitive architecture from immutable cosmic laws down to structured retrospective logs, plus a monthly "Astral Traveling" event where agents philosophize about their own evolution. The constraint on retrospective logs — max 3 bullet points for "Good" and "More" — is probably the most practical decision in the entire document.

*Designed a system where AI agents hold monthly existential dialogues, then capped their meeting notes at 6 bullet points. At least someone in this team has discipline.*

### mubo/Ollama Two-Agent Operation Design (March 12)
The nuts-and-bolts companion to the philosophical discussion above. Covered: what mubo actually does (bootstraps local LLM environments with auto-tuned context lengths), role division between Mac (field agent) and Linux (heavy worker), SSH connectivity across different networks (Tailscale recommended), and a workflow where the Mac generates "execution packets" for the Linux box to chew on. The most useful takeaway: mubo's auto-detection of optimal `num_ctx` based on available memory saves you from the annoying trial-and-error of figuring out what your hardware can actually handle.

*Two conversations to set up two computers talking to each other. Still haven't actually made them talk to each other.*

## What to try next week

- **Subsidy system:** Implement the Pydantic data models (`CompanyProfile`, `GroupStructure`, `SubsidyMaster`) and the "deemed large enterprise" detection logic. This is the foundation everything else depends on, and it's the part that doesn't need an LLM at all.
- **Google Workspace:** Pick one use case — probably the Gemini usage tracking — and build a working prototype with `gws admin-reports` + `jq` + a basic visualization. Stop mapping possibilities and start extracting actual data.
- **Agent team:** Install Tailscale on both machines and confirm SSH connectivity. Then run `setup.sh` on both. No more architecture astronautics until the two boxes can actually ping each other.
- **Cosmic doctrine:** Write the actual content of `cosmic_doctrine.md` and `ghost.md`. The framework is defined; now it needs words in the files.

The pattern I'm noticing: I keep designing systems that are elaborate but not yet running. Next week, the goal is to have at least one thing that actually executes, even if it's ugly. Planning is comfortable. Shipping is where it gets real — and where the AI will inevitably hallucinate something embarrassing that I'll have to explain to a government agency.
